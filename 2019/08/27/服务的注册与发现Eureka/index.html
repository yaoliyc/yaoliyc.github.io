High Availability, Zones and Regions

Eureka server没有后端存储，所有注册的服务示例必须发送心跳保存注册信息的更新。客户端也在内存中保存Eureka注册信息。

默认情况下，每个Eureka服务端同时也是一个客户端需要一个地址来注册。如果不提供，服务也可以正常运行但日志中会有许多提示无法注册。

独立模式

两个缓存（客户端和服务器）和心跳的组合使得独立的Eureka服务器能够非常适应故障，只要有某种监视器或弹性运行时（例如Cloud Foundry）使其保持活动状态。在独立模式下，您可能更愿意关闭客户端行为，以便它不会继续尝试并且无法访问其对等方。以下示例显示如何关闭客户端行为：

Peer Awareness

Eureka可以通过运行多个实例并要求它们相互注册，可以使Eureka更具弹性和可用性

application.yml (Two Peer Aware Eureka Servers).

上述示例中，我们在两个hosts上运行同样的服务中不同的Spring profiles。可以在单个主机上通过配置/etc/hosts解析这些主机名来测试peer awareness。事实上eureka.instance.hostname不是必须的，默认通过java.net.InetAddress来获取。

可以将多个对Peer添加到系统中，并且只要它们通过至少一个edge彼此连接，它们就会在它们之间同步注册。如果Peer在物理上是分开的（在数据中心内或在多个数据中心之间），那么系统原则上可以存在“split-brain”类型的故障。您可以向系统添加多个对等体，只要它们彼此直接连接，它们就会在它们之间同步注册。

When to Prefer IP Address

在某些时候，Eureka对服务发布IP地址而不是主机名。设置eureka.instance.preferIpAddress为true,应用使用IP地址而不是主机名想Eureka注册。

Securing The Eureka Server

可以通过添加spring-boot-starter-security为Eureka服务提供安全防护。默认情况，当classpath中出现Spring Security时将要求每次向应用程序发出请求时都会发送一个有效的CSRF令牌。Eureka客户端通常不会拥有有效的跨站点请求伪造（CSRF）令牌，您需要为/ eureka / **端点禁用此要求。示例：
